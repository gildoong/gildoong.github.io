지난 글에서는 카프카가 왜 탄생했는지를 살펴봤다. 이번에는 카프카를 실제로 다룰 때 반드시 알아야 하는 핵심 개념들을 정리한다. 이 개념들은 카프카 경험자를 면접에서 검증할 때 빠짐없이 등장하는 내용이자, 실무에서 성능 좋은 컨슈머와 프로듀서를 설계하기 위한 기초 체력이기도 하다.

다루는 순서는 다음과 같다.

1. 클러스터와 브로커
2. 토픽과 파티션
3. 오프셋
4. 프로듀서와 컨슈머, 컨슈머 그룹
5. 이벤트·메시지·레코드 용어 정리
6. 로그와 세그먼트
7. 레플리케이션 (리더, 팔로워, 레플리카, ISR)
8. 주키퍼와 KRaft

## 1. 클러스터와 브로커

카프카는 분산 이벤트 스트리밍 플랫폼이다. 하나의 서버가 아니라 여러 서버가 뭉쳐서 하나의 시스템처럼 동작하는데, 이 여러 서버의 묶음을 **카프카 클러스터**라고 부른다.

- **클러스터:** 카프카 프로세스들의 집합. 실무에서는 카프카 서버들의 집합이라고 부른다.
- **브로커:** 클러스터를 구성하는 카프카 프로세스, 혹은 그 프로세스가 동작하는 서버 한 대. 운영 환경에서는 서버와 프로세스가 1대1로 매핑되는 경우가 일반적이라 브로커를 서버 한 대로 이해해도 무방하다.

한 대의 서버 위에 여러 카프카 프로세스를 띄워서 클러스터를 구성하는 것도 기술적으로는 가능하다. 다만 이는 로컬 테스트나 예산이 부족한 베타 환경에서 쓰는 방식이고, 프로덕션 환경에서는 서버 한 대당 프로세스 하나씩 배치하는 구성이 일반적이다.

클러스터를 구성하는 이유는 크게 두 가지다.

| 목적 | 설명 |
|------|------|
| 수평 확장 | 트래픽이 늘어나면 브로커를 추가로 투입해서 용량을 쉽게 늘릴 수 있다 |
| 고가용성 | 레플리케이션을 통해 일부 브로커에 장애가 나도 나머지 브로커로 정상 동작을 유지한다 |

## 2. 토픽과 파티션

### 토픽

토픽은 메시지가 분류되어 저장되는 카테고리, 즉 주제다. 특정 관심사로 묶고 싶은 이벤트가 생기면 그 이벤트를 위한 토픽을 하나 만든다고 생각하면 된다.

예를 들어 링크드인에서 사용자가 프로필을 업데이트했다면 그 이벤트를 저장할 토픽이 하나 필요하다. 백엔드 애플리케이션에서 치명적인 로그만 따로 모으고 싶다면 그것을 위한 별도의 토픽을 만들 수 있다. 하나의 카프카 클러스터 안에는 필요한 만큼 여러 개의 토픽을 만들 수 있다.

### 파티션

파티션은 토픽 안에서 병렬 처리를 위해 나누는 단위다. 토픽 하나를 여러 개의 파티션으로 쪼개면 여러 컨슈머가 동시에 병렬로 메시지를 처리할 수 있어 처리량이 올라간다.

파티션의 핵심 특성은 다음과 같다.

- 메시지는 반드시 하나의 파티션에만 저장된다. 여러 파티션에 중복 저장되지 않는다.
- 이뮤터블하다. 한번 저장된 메시지는 수정되거나 삭제되지 않는다.
- 어펜드 방식으로만 저장된다. 항상 끝에만 추가되고, 중간에 끼워 넣는 방식은 없다.
- 컨슈머가 메시지를 읽었다고 해서 그 메시지가 바로 삭제되지 않는다. 리텐션 기간이 지나야 삭제된다.

### 순서 보장의 범위

가장 헷갈리기 쉬운 부분이다. 토픽 전체가 선입선출로 동작하는 것이 아니라, **파티션 각각의 내부에서만 순서가 보장**된다.

| 범위 | 순서 보장 |
|------|-----------|
| 같은 파티션 안 | ✅ 보장됨 |
| 서로 다른 파티션 사이 | ❌ 보장 안 됨 |

따라서 반드시 순서대로 처리되어야 하는 이벤트가 있다면, 프로듀싱 단계에서 그 이벤트들이 항상 같은 파티션으로 가도록 설정해야 한다.

## 3. 오프셋

오프셋은 파티션 안에 저장된 메시지의 순서를 나타내는 번호다.

- 각 파티션마다 독립적으로 0번부터 순차적으로 증가한다.
- 컨슈머는 자신이 몇 번 오프셋까지 읽었는지를 추적하면서 다음에 읽어야 할 메시지를 판단한다.

이 개념이 중요한 이유는 컨슈머의 동작 방식 자체가 오프셋 관리에 기반하고 있기 때문이다.

```
파티션 0:  [msg0] [msg1] [msg2] [msg3] [msg4] ...
오프셋:      0      1      2      3      4
                                 ↑
                         컨슈머가 읽은 위치 (오프셋 2까지 완료)
                         → 다음은 오프셋 3부터 읽는다
```

## 4. 프로듀서와 컨슈머

### 프로듀서

프로듀서는 애플리케이션에서 발생한 이벤트를 토픽에 저장하는 주체다. 별도의 독립된 프로듀서 애플리케이션이 존재한다기보다, 백엔드 애플리케이션 자체가 비동기로 처리하고 싶은 이벤트가 생겼을 때 그 역할을 겸하는 경우가 일반적이다.

메시지가 어떤 파티션에 저장될지는 **파티셔너**가 결정한다. 파티셔너의 세부 동작 방식은 프로듀서 편에서 자세히 다룰 예정이다.

### 컨슈머

컨슈머는 토픽에 저장된 이벤트를 가져와 목적에 맞게 처리하는 주체다. 앞서 설명한 오프셋을 기준으로 자신이 어디까지 읽었는지를 추적하며 동작한다.

### 컨슈머 그룹

컨슈머 그룹은 동일한 목적으로 묶인 컨슈머들의 집합이다. 카프카에서 가장 강력한 개념 중 하나다.

| 특성 | 내용 |
|------|------|
| 파티션 분담 | 같은 그룹 내 컨슈머들이 파티션을 나누어 맡는다 |
| 최적 구성 | 파티션 수 = 컨슈머 수일 때 1대1 매칭으로 처리량이 가장 좋다 |
| 초과 컨슈머 | 컨슈머 수가 파티션 수보다 많으면 초과분은 아무 일도 하지 않는다 |
| 오프셋 관리 | 컨슈머 개별이 아닌 **그룹 단위**로 관리된다. 특정 컨슈머가 죽어도 그룹 내 다른 컨슈머가 이어받는다 |
| 그룹 간 독립 | 서로 다른 컨슈머 그룹은 같은 토픽을 읽더라도 오프셋을 완전히 독립적으로 관리한다 |

> 예를 들어 광고 시스템 컨슈머 그룹과 뉴스피드 컨슈머 그룹이 같은 토픽을 각자의 속도와 시점으로 처음부터 끝까지 읽을 수 있다.

모든 컨슈머는 반드시 하나의 컨슈머 그룹에 속해야 한다. 단독으로 동작시키고 싶다면 컨슈머 하나짜리 그룹을 만들면 된다.

## 5. 이벤트, 메시지, 레코드: 같은 대상, 다른 관점

세 용어 모두 결국 같은 대상을 가리키지만 강조하는 관점이 다르다. 실무에서도 엄격하게 구분해서 쓰지는 않는다.

| 용어 | 강조하는 관점 |
|------|---------------|
| 이벤트 | 비즈니스 관점. 소스 애플리케이션에서 발생한 **사실**이라는 점을 강조 |
| 메시지 | 전달 관점. 프로듀서에서 컨슈머로 **전달**된다는 점을 강조 |
| 레코드 | 카프카 내부 관점. **저장**되는 데이터 그 자체를 강조 |

## 6. 로그와 세그먼트

### 로그

하나의 파티션에 순서대로 어펜드되는 이뮤터블한 메시지의 흐름 전체를 카프카에서는 **로그**라고 부른다. 일반적인 애플리케이션 로그와는 다른 개념이라는 점에 유의해야 한다.

### 세그먼트

로그를 실제로 디스크에 저장할 때는 하나의 거대한 파일이 아니라 여러 개의 파일로 쪼개서 관리한다. 이 파일 단위를 **세그먼트**라고 부른다.

세그먼트 하나는 다음 세 종류의 파일로 구성된다.

| 파일 | 역할 |
|------|------|
| `.log` | 실제 메시지가 저장되는 파일 |
| `.index` | 로그 파일을 빠르게 조회하기 위한 인덱스 파일 |
| `.timeindex` | 시간 기준으로 조회하기 위한 인덱스 파일 |

파일명 앞에 붙는 숫자는 해당 세그먼트의 첫 메시지 오프셋을 의미한다. 예를 들어 파일명이 `00000000000000000123`이라면 이 세그먼트의 첫 메시지 오프셋이 123이라는 뜻이다.

세그먼트를 나누는 이유는 관리 편의성 때문이다.

- 로그를 하나의 파일로 관리하면 파일이 지나치게 커져서 입출력과 관리, 삭제가 모두 느려진다.
- 일정 크기(기본 1GB)나 일정 시간(기본 7일)을 넘기면 새로운 세그먼트가 생성된다.
- 가장 최근에 쓰이는 세그먼트를 **액티브 세그먼트**라 부른다. 새로운 메시지는 액티브 세그먼트에만 추가된다.
- 리텐션 기간이 지나면 가장 오래된 세그먼트부터 통째로 삭제된다.

## 7. 레플리케이션: 리더, 팔로워, 레플리카, ISR

레플리케이션은 각 파티션의 데이터를 여러 브로커에 복제해서, 일부 브로커에 장애가 나더라도 메시지 유실 없이 카프카가 정상 동작하도록 하는 기능이다.

| 용어 | 의미 |
|------|------|
| 리더 | 해당 파티션에 대한 모든 읽기·쓰기를 담당하는 브로커. 파티션당 하나만 존재 |
| 팔로워 | 리더로부터 폴링 방식으로 데이터를 계속 복제해오는 브로커 |
| 레플리카 | 원본과 복제본을 가진 브로커 전체. 리더와 팔로워 모두 포함 |
| ISR (In-Sync Replica) | 리더와 정상적으로 동기화되고 있는 레플리카들의 집합 |

동작 흐름은 다음과 같다.

```
프로듀서 → 리더(브로커 A)에 데이터 쓰기
              ↓ 폴링으로 복제
          팔로워(브로커 B), 팔로워(브로커 C)

컨슈머 → 리더(브로커 A)에서 데이터 읽기

브로커 A 장애 발생
→ ISR 내 팔로워 중 하나(브로커 B)가 새 리더로 선출
→ 서비스 중단 없이 계속 동작
```

### 레플리케이션 팩터

토픽을 생성할 때 원본을 포함해 복제본을 몇 개까지 둘지 설정하는 값이다.

| 팩터 | 특성 |
|------|------|
| 1 | 복제본이 없는 셈. 장애 시 데이터 유실 위험 |
| 2 | 두 브로커 동시 장애 가능성이 있어 다소 불안 |
| **3** | **가장 널리 쓰이는 값.** 안정성과 용량 부담의 균형이 좋다 |
| 4 이상 | 절대 유실되면 안 되는 데이터에 한해 고려. 저장 용량과 복제 지연 부담 증가 |

레플리케이션 팩터는 클러스터의 브로커 개수보다 클 수 없다.

### 예시로 이해하기

- **파티션 1개, 레플리케이션 팩터 3:** 파티션 0번 하나에 대해 세 브로커가 각각 레플리카를 가진다.
- **파티션 2개, 레플리케이션 팩터 3:** 파티션 0번과 파티션 1번 각각에 대해 세 개씩의 레플리카가 만들어진다. 파티션마다 리더가 별도로 존재하므로 프로듀서는 어느 파티션에 쓰느냐에 따라 접속하는 브로커가 달라진다.

이런 구조 덕분에 특정 브로커 하나가 죽어도, 그 브로커가 맡고 있던 파티션의 리더 역할이 ISR 내 다른 브로커로 넘어가면서 서비스는 끊기지 않고 계속된다.

## 8. 주키퍼와 KRaft

카프카는 분산 시스템을 안정적으로 유지하기 위한 메타데이터 관리와 조정(coordination) 작업이 필요하다. 초기 카프카는 이 역할을 **주키퍼(ZooKeeper)** 라는 별도의 클러스터에 맡겼다.

다만 주키퍼 클러스터를 따로 운영해야 한다는 점에서 운영 복잡도가 높아지고 확장에도 제약이 있었다. 이런 한계를 해소하기 위해 등장한 것이 **KRaft**다.

| 구분 | 주키퍼 방식 | KRaft 방식 |
|------|------------|------------|
| 메타데이터 관리 | 별도의 주키퍼 클러스터 | 카프카 클러스터 내부 노드가 직접 담당 |
| 운영 복잡도 | 높음 (두 클러스터 관리) | 낮음 (단일 클러스터) |
| 확장성 | 제약 있음 | 개선됨 |
| 최신 버전 기본값 | ❌ | ✅ |

최신 버전의 카프카에서는 KRaft 사용이 기본이며, 기존에 주키퍼 기반으로 운영하던 클러스터는 버전을 올릴 때 KRaft로 전환하는 작업이 필요하다.

## 마무리

이번 글에서 다룬 개념을 한 줄씩 다시 정리하면 다음과 같다.

| 개념 | 한 줄 요약 |
|------|-----------|
| 클러스터 / 브로커 | 클러스터는 여러 브로커의 묶음, 브로커는 카프카 프로세스 한 단위 |
| 토픽 / 파티션 | 토픽은 메시지의 카테고리, 파티션은 병렬 처리를 위해 토픽을 쪼갠 단위 |
| 순서 보장 | 파티션 내부에서만 보장됨 |
| 오프셋 | 파티션 안에서 메시지의 순번, 컨슈머가 읽은 위치 추적에 사용 |
| 컨슈머 그룹 | 파티션을 나눠 맡고, 그룹마다 오프셋을 독립적으로 관리 |
| 이벤트 / 메시지 / 레코드 | 같은 대상을 가리키는 관점의 차이 |
| 로그 / 세그먼트 | 로그는 파티션에 쌓이는 메시지 흐름, 세그먼트는 그 로그를 나눠 저장하는 파일 단위 |
| 레플리케이션 | 리더·팔로워·ISR 구조로 장애에도 데이터를 지키고 서비스를 유지 |
| 주키퍼 / KRaft | 메타데이터 관리 역할, 현재는 KRaft로 전환 중 |

> 다음 글에서는 프로듀서 내부 동작을 자세히 다룬다. 오늘 정리한 토픽, 파티션, 오프셋, 컨슈머 그룹의 기본 개념이 이후 내용을 이해하는 데 기반이 되므로, 헷갈리는 부분이 있다면 다시 한번 짚고 넘어가는 편이 좋다.
