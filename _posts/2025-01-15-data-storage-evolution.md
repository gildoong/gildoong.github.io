데이터 엔지니어링을 공부하다 보면 Data Warehouse, Data Lake, Lakehouse라는 단어를 자주 만난다.  
처음엔 "저장소 종류가 왜 이렇게 많아?"였는데, 공부해보니 각각이 이전 것의 한계를 해결하기 위해 등장했다는 걸 알게 됐다.  
순서대로 정리해봤다.

---

## 먼저 — 왜 저장소가 진화했나

기업이 커질수록 부서마다 데이터를 따로 쌓기 시작한다.  
마케팅 팀은 마케팅 DB, 영업 팀은 영업 DB, 물류 팀은 물류 DB.

이게 쌓이면 **사일로(Silo) 현상**이 생긴다.

- 같은 고객 데이터인데 부서마다 숫자가 다름 → 데이터 신뢰도 하락
- 전사 관점의 분석이 필요한데 데이터가 흩어져 있어 느린 의사결정
- 저장소가 많아질수록 유지보수 비용도 따라서 증가

이 문제를 해결하려는 시도가 저장소의 진화로 이어진다.

---

## 1. Data Warehouse — 분석을 위한 중앙 집중식 저장소

### 왜 만들어졌나

흩어진 데이터를 한 곳에 모아서 분석하자는 아이디어에서 출발했다.  
OLTP 시스템(운영 DB)에서 데이터를 뽑아와, 분석에 최적화된 형태로 변환해서 저장한다.

### 어떻게 작동하나

Data Warehouse는 **OLAP(Online Analytical Processing)** 방식으로 설계된다.

OLTP와 OLAP의 차이를 먼저 이해해야 한다.

| 구분 | OLTP | OLAP |
|---|---|---|
| 목적 | 실시간 트랜잭션 처리 | 대량 데이터 분석 |
| 저장 방식 | 행(Row) 기반 | 열(Column) 기반 |
| 예시 | 배민 주문, 은행 이체 | 월별 매출 집계, AI 학습 |
| 강점 | 한 행을 빠르게 읽고 씀 | 특정 컬럼만 골라 읽어 분석 빠름 |
| 약점 | 대량 집계 시 I/O 병목 | 실시간 쓰기에 부적합 |

OLTP는 새 주문이 들어오면 해당 행 하나만 처리하면 되니 빠르다.  
반면 "이번 달 전체 매출"을 알고 싶다면 모든 행을 다 읽어야 해서 부하가 크다.

OLAP은 같은 종류의 데이터(예: 매출액)를 세로 기둥 단위로 모아두기 때문에,  
매출액 컬럼만 쏙 뽑아 읽을 수 있고 같은 타입끼리 모여 있어 압축률도 높다.

### 한계

Data Warehouse는 **정형 데이터만 다룰 수 있다.**  
스키마를 미리 정의하고, 그 형식에 맞는 데이터만 넣을 수 있다(Schema-on-Write).  
이미지, 로그, JSON 같은 비정형 데이터는 넣기 어렵다.  
또한 대용량 데이터를 저장하면 비용이 급격히 올라간다.

## 2. Data Lake — 일단 다 넣자

### 왜 만들어졌나

빅데이터 시대가 오면서 정형 데이터 외에도 로그, 이미지, 영상, JSON 등  
비정형 데이터가 폭발적으로 늘어났다. Data Warehouse로는 감당이 안 됐다.

그래서 나온 아이디어가 **"원시 상태 그대로 일단 다 쌓자"** 였다.  
S3, Azure Blob 같은 객체 스토리지 기반으로 무한에 가까운 확장이 가능하고,  
형식을 가리지 않으니 어떤 데이터든 넣을 수 있다(Schema-on-Read).

### 구현의 4가지 핵심

1. **확장성** — 객체 스토리지 기반으로 용량 걱정 없이 늘릴 수 있다
2. **데이터 접근성** — 메타데이터 카탈로그로 원하는 데이터를 검색할 수 있게 한다
3. **보안** — 암호화 및 IAM 정책으로 데이터를 보호한다
4. **비용 관리** — 자주 쓰지 않는 데이터는 저렴한 스토리지 계층으로 이동한다

### 한계 — 데이터 늪(Data Swamp)

마구잡이로 넣다 보면 시간이 지날수록 무슨 데이터가 어디 있는지 알 수 없게 된다.  
데이터 품질 보장도 없고, ACID 트랜잭션도 지원하지 않는다.  
신뢰도와 사용 속도가 떨어지면서 결국 **데이터 늪**이 된다.

---

## 3. Lakehouse — 둘의 장점을 합치자

### 왜 만들어졌나

Data Lake의 유연한 저장 + Data Warehouse의 성능과 관리 기능을 동시에 원하는 수요가 생겼다.  
특히 AI/ML 모델은 정형·비정형 데이터를 모두 필요로 하는데,  
이걸 따로 관리하면 파이프라인이 복잡해지는 문제가 있었다.

### 어떻게 작동하나

Lakehouse는 **한 곳에 모든 데이터를 저장**하되,  
Delta Lake, Apache Iceberg 같은 테이블 포맷을 얹어서  
ACID 트랜잭션, Time Travel, 스키마 관리 등 Warehouse 수준의 기능을 제공한다.

Parquet 같은 고효율 컬럼형 포맷으로 데이터를 저장하기 때문에  
AI 모델이나 분석가가 빠르게 데이터를 추출할 수 있다.

### Lakehouse 도입 시 고려할 것

1. **워크로드 진단** — 트랜잭션이 많은가, 분석·리포팅이 많은가
2. **저장 방식 결정** — 트랜잭션 중심이면 행 기반, 분석 중심이면 열 기반
3. **하이브리드 설계** — 대부분의 기업은 앞단 OLTP로 처리하고, 데이터를 캡처해서 분석계로 넘기는 파이프라인을 함께 운영한다
4. **스키마 전략** — OLTP는 정규화로 무결성을, OLAP은 비정규화/스타 스키마로 조회 속도를 챙긴다
5. **최적화** — 압축(Compression)과 파티셔닝을 적극 활용한다

## 정리 — 세 저장소 한눈에 비교

| 구분 | Data Warehouse | Data Lake | Lakehouse |
|---|---|---|---|
| 등장 배경 | 데이터 사일로 해소 | 비정형 데이터 폭증 | 둘의 한계 동시 해결 |
| 저장 데이터 | 정형 데이터만 | 정형 + 비정형 모두 | 정형 + 비정형 모두 |
| 스키마 | 미리 정의 (Schema-on-Write) | 읽을 때 정의 (Schema-on-Read) | 유연하게 관리 |
| 성능 | 분석 빠름 | 분석 느릴 수 있음 | 분석 빠름 |
| 트랜잭션 | 지원 | 미지원 | 지원 (Delta Lake 등) |
| 비용 | 높음 | 낮음 | 중간 |
| 대표 도구 | BigQuery, Redshift, Snowflake | S3 + Hadoop | Databricks, Delta Lake |

---

처음엔 "그냥 DB 아닌가?" 싶었는데,  
각 저장소가 이전 것의 한계를 보완하며 등장했다는 맥락을 알고 나니 훨씬 자연스럽게 이해됐다.  
데이터 엔지니어링 파이프라인을 설계할 때 어떤 저장소를 선택할지는 결국 **워크로드의 특성**에 달려 있다는 것도.
