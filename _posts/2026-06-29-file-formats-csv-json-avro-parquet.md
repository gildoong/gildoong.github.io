데이터 엔지니어는 여러 소스에서 디지털 데이터를 수집·통합하거나 변환하고 저장하는 일을 한다.

이 과정에서 디스크에 데이터를 쓰거나 읽는 작업이 수반된다. 원격 서버에서 API로 데이터를 가져오는 것부터 테이블에 데이터를 추가하는 것까지, 물리적 디스크에 저장된 디지털 레코드와 항상 상호작용한다. 대부분의 경우 HDD나 SSD 같은 raw 장치를 직접 다루지 않고, **파일 추상화(file abstraction)** 를 통해 작업한다.

모든 파일은 구성 방식이 다르며 그에 따라 최적화된 읽기·쓰기 방식도 다르다. 그렇기에 DE는 데이터가 어떻게 구성되며, 최적의 읽기·쓰기 작업이 무엇인지 이해해야 한다.

## 텍스트 기반 포맷 vs 바이너리 포맷

디스크에 저장된 모든 데이터는 0과 1의 연속이다. 이 0과 1을 어떻게 해석하느냐에 따라 텍스트 기반 포맷과 바이너리 포맷이 구분된다.

**텍스트 기반 포맷**은 문자 인코딩에 의존한다. `A`, `9`, `!` 같은 모든 문자에 ASCII나 유니코드 같은 표준으로 고유한 숫자 코드를 부여한다. 다만 숫자 처리에 있어서 매우 비효율적이다. `256`을 저장한다면 각 숫자에 대한 이진 코드를 따로 저장하기 때문에 더 많은 공간과 추가 변환 단계가 필요하다. 대신 사람이 읽을 수 있다는 장점이 있다.

**바이너리 포맷**은 사람이 읽을 수 있는지와 무관하게 기계의 효율성을 위해 설계되었다. 문자열은 여전히 인코딩(UTF-8)되지만, 그 외 데이터 타입은 원시 이진 형태로 저장된다. 숫자 `256`은 16비트 정수 기준으로 `00000001 00000000`으로 저장된다. CPU가 이진 값을 직접 해석할 수 있어 **컴팩트하고 처리 속도가 빠르다.**

## CSV

CSV 파일을 텍스트 편집기로 열어보면 줄바꿈 문자(`
`)로 구분된 텍스트를 볼 수 있다. `,`은 열 구분, 한 줄이 하나의 행이다.

이 단순한 구조 덕분에 스프레드시트부터 관계형 DB까지 거의 모든 데이터 애플리케이션이 CSV를 이해할 수 있어 널리 사용된다. **호환성과 사용 편의성**이 최대 강점이다.
<img width="612" height="198" alt="image" src="https://github.com/user-attachments/assets/c3b60f23-06d6-4515-aac6-a3512b34d097" />


### CSV의 한계

**1. 스키마와 데이터 타입 정의 불가**

CSV에는 스키마를 정의하거나 데이터 타입을 강제하는 메커니즘이 없다. 데이터를 파싱하고 해석하는 책임이 읽기 시스템에 있다. `"1"`이 숫자가 아닌 문자열이어야 하는 경우처럼 치명적인 오류의 원인이 된다.

**2. 표준 부재로 인한 파이프라인 취약성**

- 구분자 충돌: 데이터 필드 자체에 쉼표가 있다면 구조가 깨질 수 있다.
- 따옴표 처리 불일치: 필드를 언제 따옴표로 감쌀지 보편적 규칙이 없다.
- 인코딩 불일치: UTF-8이 권장 방식이지만 보장되지 않아 데이터 손상으로 이어질 수 있다.
<img width="570" height="212" alt="image" src="https://github.com/user-attachments/assets/46838a11-8b51-469e-9ac5-f98a599df318" />


**3. 대규모 분석에 부적합**

CSV는 행 지향 텍스트 포맷이라 대규모 분석에 비효율적이다. 전체 행을 읽고 처리해야 하기 때문에 과도한 I/O가 발생한다.

**4. 분할 가능성과 압축의 트레이드오프**

압축하지 않은 CSV 파일은 **분할 가능(splittable)** 하여 병렬 처리가 가능하다. 파일의 각 줄이 완전하고 독립적인 레코드를 나타내기 때문이다.

그러나 Gzip처럼 **분할 불가능한 압축 알고리즘**을 사용하면, 파일 전체가 하나의 연속된 압축 스트림이 된다. 처리 엔진이 작업을 나눌 수 없게 되는 것이다. CSV는 파일 포맷, Gzip은 압축 방식임을 기억하자(`data.csv.gz`).
<img width="1330" height="432" alt="image" src="https://github.com/user-attachments/assets/1e491598-cee1-4f77-9583-31998c29dfc8" />


**5. 중첩 및 반복 데이터 미지원**

중첩(nested)되거나 반복되는 데이터 구조를 지원하지 않는다. 한 사람이 여러 물건을 주문한 경우, 주문번호와 고객명이 중복으로 작성될 수밖에 없다.

```
주문번호, 고객, 상품명, 수량
001, 갈둥, 키보드, 1
001, 갈둥, 마우스, 2
```

## JSON

JSON은 텍스트 기반 데이터 교환 포맷으로, 사람이 읽을 수 있는 키-값 쌍 구조로 이루어져 있다. 중첩 객체와 배열을 기본적으로 지원해, CSV가 표현하지 못하는 복잡하거나 반정형 데이터를 표현하는 데 적합하다.

```json
[
  {
    "이름": "갈둥",
    "나이": 28,
    "주소": { "도시": "창원", "구": "성산구" },
    "스킬": ["SQL", "Python", "Spark"]
  },
  {
    "이름": "철수",
    "나이": 30,
    "주소": { "도시": "서울" }
  }
]
```

CSV였다면 스킬 개수를 예측할 수 없어 컬럼을 여러 개 두어야 하지만, JSON은 유연한 구조로 자연스럽게 담아낸다.

### JSON 데이터 타입

| 타입 | 예시 |
|------|------|
| Boolean | `true`, `false` |
| Null | `null` |
| String | `"hello"` |
| Number | `123`, `3.14`, `2.5e-2` |
| Object | `{"name": "갈둥", "age": 28}` |
| Array | `[1, "apple", true, null]` |

미리 정의된 스키마가 필요 없기 때문에 데이터 구조를 쉽게 변경하거나 복잡한 관계를 표현할 수 있다. CSV의 타입 정의 불가와 중첩 구조 미지원 문제를 해결한다.

### JSON의 한계

**1. 중복성**

모든 레코드가 속성 키를 반복한다. 레코드가 수백만 개라면 키 문자열만으로도 엄청난 용량이 낭비된다.

```json
{"이름": "갈둥", "나이": 28}
{"이름": "철수", "나이": 30}
{"이름": "영희", "나이": 25}
```

**2. 느린 파싱**

객체와 배열이 서로 중첩될 수 있어 파서가 재귀적 방식으로 구조를 탐색해야 한다. 중첩 단계마다 파서가 각 레이어의 상태를 관리해야 하므로 계산 오버헤드가 커진다.

```
객체 발견 → 안을 읽어
→ 키: "이름", 값: "갈둥" (끝)
→ 키: "주소", 값: 또 객체네?
  → 안을 읽어  ← 똑같은 작업을 다시 함
  → 키: "도시", 값: "창원" (끝)
  → 키: "구", 값: "성산구" (끝)
```
<img width="800" height="476" alt="image" src="https://github.com/user-attachments/assets/c36cd1c8-b50d-44c5-89bd-d4c9de105fff" />


**3. 스키마 없음**

CSV와 마찬가지로 스키마 강제를 기본으로 지원하지 않는다. 외부 표준인 JSON Schema를 활용하면 검증이 가능하다.

**4. 분석에 부적합**

CSV처럼 같은 레코드의 필드들이 서로 가까이 저장되어 분석에 적합하지 않다.

**5. 병렬 처리 어려움**

표준 JSON 파일은 **하나의 연속된 문서**로 취급된다. 올바르게 파싱하려면 여는 괄호 `[`부터 닫는 괄호 `]`까지 전체 구조를 읽어야 한다.

반면 **NDJSON(Newline Delimited JSON)** 은 각 줄이 완전하고 독립적인 JSON 객체여서 여러 워커가 병렬로 처리할 수 있다.

```json
// NDJSON 예시
{"이름": "갈둥", "나이": 28}
{"이름": "철수", "나이": 30}
{"이름": "영희", "나이": 25}
```

## Apache Avro

Apache Hadoop 생태계에서 탄생한 데이터 직렬화 프레임워크다. 행 지향 포맷이며, 언어에 독립적인 스키마 정의를 지원한다. JSON보다 더 많은 데이터 타입을 지원하고 Union, Enum, Map 같은 복합 타입도 포함된다.

스키마를 진화시킬 수 있는 데이터 교환 및 직렬화에 최적화되어 있어, **Apache Kafka를 활용한 스트리밍 데이터 파이프라인의 지배적인 포맷**이다.

> **직렬화(Serialization)란?**
> 메모리에 있는 데이터 구조를 저장하거나 전송할 수 있는 형태로 변환하는 것이다.
> ```
> 메모리의 객체  →  직렬화  →  바이트 스트림 (저장/전송)
> 바이트 스트림  →  역직렬화  →  메모리의 객체
> ```
> Kafka로 데이터를 주고받을 때 데이터를 바이트로 변환해서 보내고, 받는 쪽에서 다시 복원하는 과정이 필요한데, Avro가 이를 효율적으로 처리한다.

### 스키마 진화(Schema Evolution)

스키마는 보통 JSON으로 정의되고, 실제 데이터는 바이너리 포맷으로 저장된다. Avro는 매 레코드마다 필드 이름을 저장하지 않고, **스키마에 정의된 순서대로 값만 저장**한다.

```
// JSON이었다면 (키를 매번 반복)
{"이름": "갈둥", "나이": 28, "도시": "창원"}

// Avro는 (값만 순서대로)
"갈둥" | 28 | "창원"
```

이 덕분에 데이터가 압축되어 저장 공간과 네트워크 전송 오버헤드가 줄어든다.

<img width="838" height="608" alt="image" src="https://github.com/user-attachments/assets/cfab4cd1-2d3a-4651-b1b4-1681560f1e80" />


**Writer 스키마 vs Reader 스키마**

- **Writer 스키마:** 데이터를 쓸 때 사용한 스키마
- **Reader 스키마:** 데이터를 읽을 때 기대하는 스키마

핵심은 이 둘이 같을 필요가 없고 **호환 가능하기만 하면 된다**는 점이다.

```
Writer가 쓸 때:  [이름, 나이, 도시]
Reader가 읽을 때: [이름, 나이, 도시, 이메일]  ← 새 필드 추가됨
```

Reader 스키마에 이메일이 추가되어도 Avro가 이 차이를 자동으로 해결한다. **파이프라인을 멈추지 않아도 스키마를 변경할 수 있다**는 것이 큰 강점이다.

<img width="922" height="324" alt="image" src="https://github.com/user-attachments/assets/efb89f73-40b4-464b-92e6-93c66c58cc6e" />


### 병렬 처리

Avro 데이터 파일은 병렬 처리가 용이하도록 설계되어 있다.

<img width="554" height="314" alt="image" src="https://github.com/user-attachments/assets/ce930343-e828-4a9f-b0c8-ac6aa3f2bae7" />


- **File Header:** 파일에 저장된 데이터의 스키마를 담은 헤더로 시작한다.
- **Data Blocks:** 헤더 다음에 일련의 데이터 블록들이 이어진다. 각 블록은 독립적으로 압축된다.
- **Sync Markers:** 각 데이터 블록 사이에는 **16바이트 sync marker**가 있다. 파일마다 고유하게 랜덤 생성된 바이트 시퀀스로, 파일 헤더에도 저장된다.

Sync marker 덕분에 Reader가 파일을 여러 부분으로 나눌 수 있고, 각 워커가 독립적인 데이터 블록들을 담당할 수 있다. Gzip으로 압축한 CSV는 하나의 덩어리이지만, Avro는 각 블록이 독립적으로 압축되어 있고 sync marker로 경계가 명확해 중간 지점부터 읽어도 블록의 시작점을 찾을 수 있다.

<img width="778" height="504" alt="image" src="https://github.com/user-attachments/assets/9c46fe80-bae4-45a8-8312-4cbac9bb7eae" />


### Avro의 단점

가장 명확한 단점은 **행 지향 포맷**이라는 점이다. 여전히 분석 워크로드에는 적합하지 않다.

## Parquet

Parquet는 **분석 워크로드를 염두에 두고 설계된 바이너리 포맷**이다.

분석 워크로드는 대부분 전체 컬럼 중 일부만 필요하기 때문에, 쿼리 성능은 주로 얼마나 많은 데이터를 건너뛸 수 있느냐에 달려 있다. Parquet는 데이터를 **하이브리드 포맷**으로 저장한다. 데이터를 Row Group이라는 단위로 수평 분할하고, 각 그룹 안에서는 컬럼 데이터를 가까이 모아 저장한다. 쿼리 엔진이 필요한 Row Group과 컬럼만 읽을 수 있어 I/O를 크게 줄일 수 있다.

<img width="786" height="458" alt="image" src="https://github.com/user-attachments/assets/9113d410-76e2-44ff-8134-6ebf73e4abe0" />


### 파일 구성 요소

**Row Group**

데이터셋의 행들을 나눈 단위다. 그 안에서는 컬럼별로 구성되며, 각 컬럼은 별도의 Column Chunk에 저장된다.

```
전체 데이터 (1000행)
├── Row Group 1 (1~300행)
├── Row Group 2 (301~600행)
└── Row Group 3 (601~1000행)
```

**Column Chunk**

Row Group 안에서 특정 컬럼 하나의 데이터 전체다. 같은 컬럼의 데이터가 연속으로 저장되기 때문에 데이터가 반복될 가능성이 높아 압축 효율이 좋다.

**Page**

Column Chunk를 더 잘게 나눈 단위로, Parquet에서 가장 작은 데이터 단위다.

- **Data Page:** 실제 데이터를 담고 있다.
- **Dictionary Page:** 딕셔너리 인코딩된 값을 담고 있다.
- **Index Page:** 빠른 데이터 조회를 위해 사용된다.

**메타데이터**

Avro처럼 Parquet도 스키마와 메타데이터를 파일 내부에 저장하는 **self-contained 포맷**이다.

- **Magic Number:** 파일의 시작과 끝에 위치한 특정 바이트 시퀀스(`PAR1`). 유효한 Parquet 파일인지 검증하는 데 사용된다.
- **FileMetadata (Footer):** 파일 끝(footer)에 저장되며 전체 행 수, 스키마, 각 Row Group 및 Column Chunk의 인코딩·압축 방식, 크기, 페이지 오프셋, **최솟값·최댓값** 등을 포함한다. 쿼리 엔진은 이 메타데이터를 활용해 불필요한 데이터를 걸러낸다.
- **PageHeader:** 페이지 데이터와 함께 저장되는 메타데이터로, 값·definition·repetition 인코딩 정보를 포함한다.

### 스키마 진화

Parquet는 기존 데이터를 다시 쓰지 않고도 스키마를 수정할 수 있다.

| 변경 유형 | 동작 |
|-----------|------|
| 컬럼 추가 | 기존 파일은 해당 컬럼 쿼리 시 null 반환 |
| 컬럼 삭제 | 클라이언트가 해당 컬럼을 선택하지 않으면 됨. 기존 데이터는 남아있지만 무시됨 |
| 컬럼 순서 변경 | 이름으로 읽기 때문에 순서 무관 |
| 컬럼 이름 변경 | **주의 필요.** 기존 컬럼 삭제 + 새 컬럼 추가와 같다. 기존 파일과 새 파일 간 데이터가 분리된다 |

Avro는 Writer/Reader 스키마 간 자동 변환이 가능하지만, Parquet는 이름 변경을 아예 별개 컬럼으로 취급한다는 차이가 있다.

### 병렬 처리

Parquet 데이터는 Row Group 단위로 수평 분할되어 있다. 처리 엔진이 서로 다른 작업이나 스레드를 할당하여 같은 파일의 여러 Row Group을 **동시에 읽을 수 있다.** 같은 Row Group 안에서도 Column Chunk 단위 병렬 읽기가 가능하다.

### Parquet의 단점

하이브리드 포맷은 분석에 이상적이지만, **쓰기 작업은 행 지향 포맷보다 느리다.** Row Group 안에서 여러 컬럼을 따로따로 써야 하기 때문이다.

| 포맷 | 쓰기 속도 | 이유 |
|------|-----------|------|
| 행 지향 (Avro) | 가장 빠름 | 한 레코드의 데이터를 연속으로 씀 |
| 하이브리드 (Parquet) | 중간 | Row Group 내에서 데이터 지역성 유지 |
| 완전 컬럼 지향 | 가장 느림 | 컬럼이 완전히 다른 파일로 분리됨 |
