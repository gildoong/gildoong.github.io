DBMS에서 SQL 쿼리를 통해 빠른 시간에 결과를 돌려주는 단순한 인터페이스 덕분에 보통 데이터베이스가 단순한 시스템이라고 하지만 사실이 아니다.
DB 엔지니어들은 가장 뛰어난 소프트웨어 엔지니어 중 하나인데, PostgreSQL, MySQL, SQLite, DuckDB, Clickhouse, BigQuery, Snowflake 같은 시스템을 만들려면, 엔지니어들이 수많은 것들을 처리해야 하기 때문이다.

데이터 저장부터 쿼리 파싱, 계획 수립 및 실행, 여러 클라이언트를 하나처럼 처리하기, ACID 보장, 그 외 수많은 측면까지. 내가 알기로는 대부분의 DBMS는 디스크, RAM, CPU 같은 물리적 자원을 제어하는 데 OS에 의존하지 않기 때문에, 엔지니어들이 해야 할 작업량은 훨씬 더 많다.

데이터 엔지니어들은 다른 일반 사용자들처럼, 보통 DBMS의 아주 저수준 세부사항을 직접 다룰 필요는 없다. 그래도 우리가 사용하는 시스템의 특성을 이해하는 것은 언제나 좋은 일이다. 우리의 주된 목표는 조직 내에서 데이터가 효율적으로 저장되고, 분석되고, 활용되도록 보장하는 것이다.

> **ACID : 데이터베이스의 4가지 약속**
> - **A** - Atomicity, 원자성: 트랜잭션은 전부 성공하거나 실패해야 한다.
> - **C** - Consistency, 일관성: 트랜잭션 전후로 데이터 무결성 규칙이 항상 유지된다.
> - **I** - Isolation, 격리성: 동시에 실행되는 트랜잭션들이 서로 간섭하지 않아야 한다.
> - **D** - Durability, 내구성: 커밋된 트랜잭션은 시스템이 죽어도 사라지지 않아야 한다.

## OLTP 및 OLAP 시스템의 작업 부하

디스크에 저장된 데이터는 기본적으로 0과 1의 연속이며, 효율적인 쓰기 및 읽기 작업을 보장하기 위해서 데이터가 기록되는 방식을 구성하고 관리하는 것은 DBMS의 책임이다.

하지만 데이터 레이아웃의 효율성은 작업 부하에 따라 평가해야 한다. 데이터베이스 작업 부하는 일반적으로 두 가지 주요 유형으로 분류할 수 있다.

**OLTP** 워크로드는 새로운 정보를 수집하며, 데이터 작업은 대부분 매번 소량의 데이터를 읽기/쓰기/수정하는 방식이다. 일반적으로 OLTP 시스템은 실제 세계의 정보를 수집하기 위해 회사에서 가장 먼저 구축된다. 웹 애플리케이션 뒤에 있는 데이터베이스나 은행 거래를 기록하는 데이터베이스가 그 예다. 이 워크로드는 백엔드 개발자처럼 회사의 제품/서비스와 밀접하게 일하는 사람들에게 더 친숙할 것이다.

<img width="630" height="238" alt="image" src="https://github.com/user-attachments/assets/2a36cb01-8c6f-4a3c-ac70-4fa4a66a1cad" />
<img width="712" height="352" alt="image" src="https://github.com/user-attachments/assets/dffbac5a-b5a2-4123-a79f-97fca82f1bfc" />




**OLAP** 워크로드는 OLTP와 다른 특성을 가진다. OLAP의 쿼리는 여러 테이블에서 데이터를 읽어야 한다(조인을 통해). 데이터를 가져온 후, 인사이트를 추출하거나 새로운 패턴을 발견하기 위해 수많은 집계 연산이 필요할 수 있다. 대부분의 경우 OLAP 워크로드는 OLTP 시스템에서 수집된 데이터를 대상으로 실행된다. 이 워크로드는 데이터 엔지니어나 분석가 같은 회사의 데이터 분석 직군에 더 친숙할 것이다.

## OLTP

대부분의 OLTP 시스템은 작업 부하를 처리하기 위해 데이터를 행 기반 방식으로 구성한다. DBMS는 단일 행의 모든 열 값을 되도록이면 한 페이지에 함께 저장하고자 한다.

데이터 기록 관점에서 볼 때, 이는 데이터를 시스템에 최대한 빠르게 기록해야 할 때 최적의 방식이다. 단일 행의 경우, 기록자는 서로 다른 열의 바이트를 순차적으로 기록해야 한다(순차 작업은 항상 빠르다고 보면 된다).

읽기 관점에서 볼 때, 데이터 구성 방식은 OLTP 워크로드에도 매우 적합하다. 일반적으로 한 행 전체를 읽어야 하는데, 한 행에 대한 모든 바이트가 함께 저장되어 있으므로 순차적인 작업이 가능하다.

## OLAP

OLAP의 경우 워크로드가 다르다. 일반적으로 OLAP 시스템은 대량의 과거 데이터를 불러와 집계 및 조인해야 하는 읽기 작업에 더 중점을 둔다. 대부분의 경우 일부 열만 스캔하면 된다. 이러한 특성 때문에 행 형식 스토리지를 사용하는 OLTP 시스템에서 OLAP 워크로드를 처리할 때 문제가 발생한다.

10개의 열로 구성된 테이블이 있다고 가정한다면, 한 행의 모든 열 값은 연속적으로 저장된다. 날짜와 매출이라는 두 열을 읽어 날짜별 매출 합계를 계산하는 OLAP 쿼리를 생각해 보자.

행 기반 저장 방식을 사용하면 전체 행을 메모리에 로드해야만 시스템이 두 열에서 데이터를 추출할 수 있다. 테이블에 100만 개의 행이 있다고 가정하면 시스템 오버헤드가 엄청날 것이다.

해결책은 데이터를 단일 열에 연속적으로 저장하는 것이다. 주요 접근 방식에는 ***컬럼 스토어 방식과 하이브리드 방식*** 두 가지가 있다.

## 컬럼 스토어

첫 번째 접근 방식은 단일 열의 데이터를 별도로 저장한다. 시스템은 날짜와 판매 데이터가 저장된 페이지에만 신경 쓰면 되므로 다른 열의 중복 데이터를 불러오는 것을 방지할 수 있다.
<img width="490" height="366" alt="image" src="https://github.com/user-attachments/assets/05b74474-3e7a-475f-90ae-8067c41a5a45" />


하지만 이 방식에서 시스템이 관리해야 할 또 다른 중요한 요소가 있는데, 바로 데이터 오프셋이다. 데이터 행이 여러 위치에 저장되기 때문에 DBMS는 페이지의 특정 열 값이 어느 행에 속하는지 알아야 하는데, 이를 위해 DSM(Decomposition Storage Model)은 각 열의 값이 동일한 길이를 갖도록 한다. 이렇게 하면 DBMS는 고정된 값 크기를 사용하여 특정 값의 오프셋을 쉽게 계산할 수 있다.

> **데이터 타입이 바이트를 결정한다.**
> <img width="774" height="524" alt="image" src="https://github.com/user-attachments/assets/555296dc-1c46-45ab-afe7-0e16fec7fc8e" />


이 저장 모델은 DBMS가 필요한 열에서만 데이터를 읽기 때문에 쿼리당 낭비되는 I/O를 줄여준다. 또한 열 값에 패턴이 나타나는 경우가 많으므로 데이터 압축률도 향상된다. Clickhouse나 Redshift에서 이 방식을 찾아볼 수 있다.

하지만 이 모델은 쓰기 작업에서 몇 가지 문제를 일으킬 수 있다. 100개의 열이 있는 테이블에 데이터를 쓸 때, DBMS는 데이터를 100개의 서로 다른 위치에 쓰기 위해 여러 단계를 거쳐야 하므로 상당한 오버헤드가 발생한다.

또한 모든 열의 데이터를 읽어야 하는 읽기 작업은 시스템에 더 많은 작업 부담을 준다. DBMS가 100곳의 데이터를 통합해야 하기 때문이다.

## Hybrid

위 문제를 해결하기 위해 하이브리드 포맷이 해결책으로 등장했다.

이 저장 모델에서는 테이블 데이터가 행 그룹으로 수평 분할되고, 각 그룹 내에서 열 데이터는 서로 인접하게 저장된다. 목표는 열 저장소에서 빠르고 효율적인 데이터 스캔을 활용하는 동시에 행 저장소의 지역성(동일한 행의 열 값은 적어도 행 그룹 내에서는 서로 가까이 위치함)을 유지하는 것이다.
<img width="510" height="630" alt="image" src="https://github.com/user-attachments/assets/f9cf8000-acec-4f89-b4bd-0aa155bb2567" />


```
┌─────────────────────────┐
│ Row Group 1             │  ← 1~100번 행 담당
│  [ColA][ColA][ColA]     │  ← A컬럼 값들 인접 저장
│  [ColB][ColB][ColC]     │  ← B,C컬럼 값들 인접 저장
│  [ColC][ColD][ColD]     │  ← D컬럼 값들 인접 저장
└─────────────────────────┘
┌─────────────────────────┐
│ Row Group 2             │  ← 101~200번 행 담당
│  [ColA][ColA][ColA]     │
│  ...                    │
└─────────────────────────┘
```

## 쓰기 작업이 많은 OLAP 시스템

최근 몇 년 동안 실시간 분석 기능을 지원하는 OLAP 시스템이 점점 더 많아지고 있는데, 대표적인 예로 BigQuery를 들 수 있다. 엔지니어들은 스트리밍 데이터 수집을 최우선으로 처리하는 스토리지 엔진을 재구축하기로 결정했으며 전체적인 접근 방식은 다음과 같다.
<img width="850" height="372" alt="image" src="https://github.com/user-attachments/assets/93d90ac9-2860-438b-8d02-0ad42e17337c" />


- 데이터는 행 저장소 형식으로 입력된다.
- 이후 사용하기 쉽도록 백그라운드에서 컬럼형/하이브리드 형식으로 변환된다.

특히 OLAP 워크로드가 실시간 데이터 수집 및 쿼리와 같은 트렌드로 끊임없이 진화하고 있는 만큼, 이미 많은 다른 시스템들이 이러한 접근 방식을 사용하고 있거나 적어도 고려하고 있을 것이라고 생각한다. 지금은 BigQuery나 Hudi만이 이러한 접근 방식을 구현하고 있는 것 같다.

## DB에서 데이터를 찾는 전략

위와 같이 저장된 데이터는 어떻게 찾아가는 것일까? 이는 데이터베이스의 작업 부하를 결정하는 중요한 요소이다.

"특정한 항목 하나를 찾아라"라는 요구를 받는 OLTP 시스템은 지도와 같은 정확한 접근 방식을 필요로 하지만, "모든 데이터에 걸쳐 몇 가지 속성을 요약하라"는 요구를 받는 OLAP 시스템은 효율적인 데이터 제거 방법을 필요로 한다.

## OLTP — B-Tree 인덱스

OLTP에서는 데이터를 찾기 위해 **B-Tree 인덱스**가 일반적으로 사용된다. 수십 년간 OLTP 데이터베이스의 성능 최적화를 위한 주요 방식이었다.

B-Tree는 정렬된 데이터를 유지하는 **자가 균형 트리 자료구조**로, 검색·삽입·삭제를 **O(log n)** 으로 수행한다. 데이터베이스를 고정 크기 페이지(예: PostgreSQL은 8KB)로 나누며, 한 번에 하나의 페이지를 읽거나 쓴다. 페이지들은 트리 구조를 형성한다.
<img width="662" height="418" alt="image" src="https://github.com/user-attachments/assets/3ed70f35-c666-4aae-9e06-d3893ade0c92" />


- **Root page**: 부모가 없으며 트리의 **진입점** 역할을 한다.
- **Internal pages**: Root와 Leaf 페이지를 **연결**하는 중간 노드.
- **Leaf pages**: 가장 아래에 있는 페이지로 자식 페이지가 없다. **모든 실제 데이터는 Leaf에 저장**된다.

> 고전적인 B-Tree는 non-leaf 페이지에도 데이터를 저장할 수 있다. 변형인 **B+ Tree**는 오직 Leaf 페이지에만 데이터를 저장한다. 이를 통해 모든 데이터 작업이 Leaf 페이지에만 집중될 수 있다. B+ Tree가 널리 채택되어 보통 B-Tree라고 불린다.

```
          [Root]    ← 진입점. "어느 방향으로 가야 하나" 안내
         /   |   \
      [In] [In] [In]         ← Internal nodes: 길잡이 역할만 함, 데이터 없음
      /      |      \
[Leaf]   [Leaf]   [Leaf]     ← 실제 데이터가 여기에만 있음
```
<img width="1224" height="374" alt="image" src="https://github.com/user-attachments/assets/74f5df71-ddea-488d-92e7-fe994525bf35" />


Internal 페이지는 실제 데이터 없이 "key 비교 → pointer 따라가기"만 담당한다. 정렬된 key 덕분에 빠르게 방향을 결정할 수 있다.

- 다른 페이지를 가리키는 **포인터**들.
- 키와 포인터는 서로 연관되어 있다: **포인터의 수 = 키의 수 + 1**
- 포인터는 **[key_left, key_right)** 범위의 키를 가진 서브트리를 가리킨다.

**실제 예시** — key1=10, key2=20, key3=30이고, id=25를 찾는다면:

```
[ptr][10][ptr][20][ptr][30][ptr]
                  ↓
          key2(20) <= 25 < key3(30)
                  ↓
           3번 pointer 따라감
```

→ 이걸 Leaf에 도달할 때까지 반복 → **O(log n)**
<img width="1018" height="432" alt="image" src="https://github.com/user-attachments/assets/e9da4d6b-77c4-47d0-aaa3-35780f3a37af" />


B-Tree에서 데이터 작업은 결국 **B-Tree에서 필요한 데이터를 찾는 것**으로 귀결된다. 읽기와 수정은 유사한 과정이다. Root에서 시작 → 키 비교 → 포인터 따라가기 → Leaf 도달. 이 단순한 반복이 B-Tree 탐색의 전부이다.
<img width="856" height="438" alt="image" src="https://github.com/user-attachments/assets/45d0c612-8ca8-4788-b034-cf552170a672" />


## OLAP — 데이터를 덜 읽는 전략

B-Tree 인덱스는 **포인트 조회와 범위 스캔 쿼리**를 빠르게 해준다. `WHERE username = "bruce_banner"` 같은 쿼리에서 인덱스는 해당 레코드가 정확히 어디 있는지 알려준다.

그러나 **OLAP에서는 조회 인덱스가 별로 도움이 되지 않는다.** 수십억 개의 행을 스캔하는 쿼리에서 성능 병목은 단일 레코드를 찾는 것이 아니라, **스토리지에서 읽고 처리해야 하는 데이터의 양을 최소화**하는 것이다. 주된 최적화 방법은 애초에 **불필요한 데이터를 최대한 읽지 않는 것**이다.

컬럼 기반 저장은 엔진이 필요한 컬럼만 읽고 나머지는 건드리지 않도록 해준다. 이 새로운 레이아웃이 도움이 되긴 하지만, 더 **세밀한 수준**에서 불필요한 데이터를 건너뛰고 싶어한다.

```
OLTP 고민:
"수억 건 중에서 id=25를 어떻게 빨리 찾지?"
→ B-Tree 인덱스로 위치를 바로 찾음

OLAP 고민:
"수십억 건을 스캔해야 하는데 어떻게 덜 읽지?"
→ B-Tree 인덱스는 의미없음, 컬럼 기반 저장 + 데이터 스킵
```

## Lightweight Metadata — Zone Map

이 과정을 가능하게 하는 메타데이터는 흔히 **"Zone Map"** 또는 **블록 수준 통계**라고 불린다. 각 데이터 청크와 그 안의 각 컬럼에 대해, 시스템은 핵심 지표 — 가장 일반적으로는 **최솟값과 최댓값** — 를 저장한다. 그 외 통계로는 null 값 개수, 고유값 개수, 또는 존재 여부 확인을 위한 **Bloom filter** 같은 정교한 자료구조가 포함될 수 있다.
<img width="512" height="504" alt="image" src="https://github.com/user-attachments/assets/1e3c8b9a-8521-45da-85ed-d5f0f6845dae" />


월별 파일로 파티셔닝된 판매 데이터 테이블에 각 파일마다 `click_counts`의 min/max를 추적하는 Zone Map이 있다고 하자. `WHERE click_counts BETWEEN 5 AND 7` 쿼리가 들어오면 쿼리 옵티마이저는 먼저 메타데이터를 스캔하고, click_counts 범위가 5~7에 해당하지 않는 블록은 즉시 버린다. 해당 범위에 속하는 블록만 디스크에서 읽어 **스캔 범위를 극적으로 줄인다.**

```
Parquet  → Row Group별 min/max 통계 내장
Iceberg  → 파일 단위 통계를 메타데이터 파일로 별도 관리
BigQuery → Google이 자체 시스템으로 이 메타데이터 관리
```

핵심: 디스크 I/O가 OLAP의 가장 큰 병목 → Zone Map으로 읽기 전에 미리 걸러내는 것이 핵심 최적화.

## Partitioning

본질적으로 **파티셔닝**은 데이터셋을 더 작은 부분으로 나누는 것이다. 궁극적인 목표는 불필요한 부분을 건너뜀으로써 **데이터 스캔을 줄이는 것**이다. 대부분의 시스템은 사용자가 더 높은 수준에서 데이터를 파티셔닝할 수 있도록 허용한다.

특정 컬럼을 지정하면 시스템이 그 값을 기준으로 테이블을 더 작은 파티션들로 나눈다. 날짜 컬럼을 지정하면 테이블이 2025-05-01, 2025-05-02, 2025-05-03 등의 파티션으로 나뉜다.
<img width="660" height="374" alt="image" src="https://github.com/user-attachments/assets/8c4fdcf6-3581-400f-9407-4d008f4457ac" />


파티셔닝은 시스템이 관련된 부분에만 작동하도록 도와준다. 쿼리에 파티션 키에 대한 **필터 조건** (예: `WHERE date=2025-05-03`)이 포함되어 있으면, 쿼리 옵티마이저는 `date=2025-05-03` 파티션만 관련이 있다고 판단하고 나머지를 완전히 무시, 즉 **"prune(가지치기)"** 할 수 있다.
<img width="818" height="426" alt="image" src="https://github.com/user-attachments/assets/41df7964-e41b-4c46-8b3b-3eeb91caf31b" />


BigQuery와 대부분의 다른 웨어하우스 시스템은 파티션을 **가상 테이블**처럼 취급한다. 한 파티션의 데이터는 다른 파티션의 데이터와 **별도로 저장**된다. 이를 통해 데이터 만료, 데이터 삽입, 데이터 삭제 같은 기능을 **파티션 단위**로 효과적으로 실행할 수 있다. 각 파티션은 연관된 메타데이터를 가지며, 엔진이 이를 데이터 pruning에 활용할 수 있다.

Snowflake는 BigQuery와 다르다. BigQuery는 사용자가 **파티션 기준 컬럼**을 직접 지정해야 하지만, Snowflake는 사용자가 아무것도 지정하지 않아도 **크기 기준으로 자동 분할**한다. Snowflake가 테이블을 **마이크로 파티션**으로 쪼개는데, 각 마이크로 파티션은 비압축 기준 **50MB ~ 500MB**의 데이터를 저장한다.
<img width="760" height="350" alt="image" src="https://github.com/user-attachments/assets/ea0f9378-ba08-494c-a969-c08233d23e0d" />


마이크로 파티션은 **하이브리드 포맷**과 유사하게 구성된다. 파티션이 행 그룹을 포함하고, 각 컬럼 데이터는 파티션 안에서 함께 저장된다. Snowflake는 데이터 관리를 용이하게 하기 위해 마이크로 파티션 내 컬럼에 대한 메타데이터를 관리한다.

스토리지 레이어를 **직접 명시적으로 관리**할 때는 데이터를 **Hive 스타일**로 구성하는 일반적인 방식을 볼 수 있다. 데이터가 폴더로 구성되는 방식이다.
<img width="508" height="342" alt="image" src="https://github.com/user-attachments/assets/2b490305-e5b9-4c04-b97c-26d4d6a1d3d3" />


- **테이블**: 각 테이블은 하나의 디렉토리를 가진다.
- **파티션**: 각 테이블은 파티션을 가질 수 있다. 각 파티션은 하나의 서브디렉토리에 해당한다.

이 방식은 단순명료하며 도입 이후 널리 채택되어 왔다. Delta Lake, Iceberg, Hudi 같은 차세대 테이블 포맷들은 사용자에게는 여전히 이 파티션 구조로 보이지만, 내부적으로는 더 **강력한 메타데이터**를 추가해 성능과 효율성을 개선한다.

```
Hive-style:
/date=2025-05-01/part-00001.parquet
→ 파일만 있음, 메타데이터 없음

Iceberg:
/date=2025-05-01/part-00001.parquet
+ /metadata/snap-001.json  ← 파일 통계, 스키마, 히스토리 등 관리
→ 같은 폴더 구조지만 메타데이터가 훨씬 풍부
```

Hive-style = 기초 뼈대. Delta Lake/Iceberg/Hudi = 같은 뼈대에 강력한 엔진을 얹은 것. DE 실무에서 Iceberg가 점점 표준이 되어가는 이유다.

## Clustering

파티셔닝처럼 클러스터링의 목표도 엔진이 불필요한 데이터를 건너뛰도록 돕는 것이다. 하지만 다른 방식으로 접근한다. 파티셔닝이 **거친 수준(coarse-grained)** 의 데이터 스킵 메커니즘을 제공하는 반면, 그 효과는 파티션 키의 cardinality(컬럼의 고유 값 수)에 의해 제한된다.

예를 들면, 날짜는 일 년에 365개뿐이지만, 유저의 수는 몇십만 명이 넘어갈 수 있기에 파티셔닝이 불가능할 수 있다.

더 세밀한 쿼리 최적화를 위해 많은 시스템이 클러스터링을 사용한다. 테이블이 파티셔닝되어 있으면 클러스터링은 파티션 내에서 관련 데이터를 함께 구성하고, 파티셔닝이 없으면 테이블 전체 수준에서 클러스터링이 이루어진다.

가장 직관적인 방법은 하나 이상의 컬럼을 기준으로 데이터를 **정렬해서 저장**하는 것이다. 이 정렬을 통해 클러스터링 컬럼에서 유사하거나 동일한 값을 가진 행들이 **co-located(같은 위치에 배치)** 되어, 디스크의 동일한 데이터 단위에 함께 쓰일 가능성이 높아진다.
<img width="884" height="530" alt="image" src="https://github.com/user-attachments/assets/77a4c71e-8b5d-4564-9ffb-3daaca4e363e" />


하지만 정렬은 **클러스터링 컬럼을 두 개 이상 지정할 때는 잘 작동하지 않는다.** 시스템은 클러스터링 컬럼의 **순서대로** 데이터를 정렬한다. 이제 쿼리가 이 정렬 순서와 맞아야 한다. Column A, 그다음 Column B 순으로 정렬한다고 하면, WHERE 필터도 반드시 이 순서를 따라야 한다: Column A 먼저, 그다음 Column B. **Column B만으로 필터링하면 정렬의 이점을 전혀 누릴 수 없다.**
<img width="928" height="418" alt="image" src="https://github.com/user-attachments/assets/b461850d-6724-4f46-891d-0f039d88b4d7" />


- File 1: device_id=100, customer_id 100~103
- File 2: device_id=101, customer_id 100~103
- File 3: device_id=102, customer_id 100~103
- File 4: device_id=103, customer_id 100~103

`device_id=100` 으로 필터링하면 File 1만 스캔하면 된다. `device_id=100 AND customer_id=102` 도 File 1만 스캔하면 된다. 하지만 `customer_id=101` 만으로 필터링하면 4개 파일 전부 스캔해야 한다.

그래서 데이터를 더 효율적으로 분산시키기 위한 **다차원 클러스터링 기법**이 등장했다. 가장 대표적인 구현이 **Z-ordering**이다.

### Z-ordering이 뭔가?

일반 정렬의 문제:

```
device_id 기준 정렬 → customer_id 단독 필터 시 모든 파일 스캔
```

Z-ordering은 **두 컬럼을 동시에 고려**해서 비슷한 값끼리 물리적으로 가깝게 배치한다:

```
일반 정렬:                    Z-ordering:
(100,100) → File 1           (100,100) (100,101) → File 1
(100,101) → File 1           (101,100) (101,101) → File 1
(100,102) → File 1           (100,102) (100,103) → File 2
(100,103) → File 1           (101,102) (101,103) → File 2
(101,100) → File 2           ...
...
```

`WHERE customer_id=101` 쿼리라면:

- 일반 정렬 → 4개 파일 전부 스캔 (비효율)
- Z-ordering → 일부 파일만 스캔 ✅ (완벽하진 않지만 훨씬 나음)

---

이 글에서는 먼저 DBMS가 단순하지 않다는 전제 아래, OLTP/OLAP 워크로드의 차이를 살펴봤다.

그 기초 위에서 두 시스템이 데이터를 어떻게 구성하는지 탐구했다. **Row-store**는 한 번에 전체 레코드를 읽고 쓰는 성능 덕분에 OLTP에 가장 적합하다. 하지만 OLAP에는 적합하지 않다.

그래서 **Column-store와 Hybrid store**가 등장한다. 마지막으로 시스템이 효율적으로 데이터를 찾는 주요 방식을 살펴봤다. OLTP는 **B-Tree 인덱스**가 핵심이고, OLAP은 **스캔하는 데이터의 양을 최대한 줄이는 것**이 핵심 방법론이다. 세 가지 주요 접근법은 **메타데이터(Zone Map), 파티셔닝, 클러스터링**이다.
