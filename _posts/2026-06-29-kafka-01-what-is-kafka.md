카프카를 처음 접하면 흔히 구조부터 파고들게 된다. 브로커가 어떻고 파티션이 어떻고 하는 이야기를 먼저 듣다 보면 정작 카프카가 어떤 문제를 풀기 위해 태어났는지는 놓치기 쉽다. 이 글에서는 구조적인 세부 사항보다 카프카가 개념적으로 무엇인지, 그리고 어떤 배경에서 탄생했는지를 먼저 정리한다. 기술이 태어난 맥락을 이해하면 그 기술을 언제 어떻게 써야 하는지에 대한 감각이 자연스럽게 잡히기 때문이다.

---

## 1. 카프카란 무엇인가

아파치 카프카 공식 홈페이지에는 다음과 같은 핵심 문구가 적혀 있다.

> Apache Kafka is an open source distributed event streaming platform.

이 한 문장 안에 카프카를 이해하는 데 필요한 다섯 가지 키워드가 모두 들어 있다.

### 오픈소스

카프카는 아파치 소프트웨어 재단(ASF)이라는 비영리 재단 아래에서 관리되는 오픈소스 프로젝트다. 누구나 참여해서 소스 코드에 기여할 수 있는 구조로 운영된다.

### 분산 시스템

한 대의 컴퓨터가 아니라 여러 대의 컴퓨터, 즉 카프카에서는 브로커라 부르는 인스턴스들이 네트워크로 연결되어 하나의 클러스터를 이루고, 그 클러스터 전체가 마치 하나의 거대한 시스템처럼 동작한다.

### 이벤트

이벤트란 소프트웨어 시스템에서 발생한 어떤 사실을 나타내는 **불변(immutable)의 데이터**를 말한다. 카프카에 데이터가 한번 저장되면 그 순간부터 그 데이터는 변하지 않는다.

예를 들어 링크드인에서 사용자가 이직 정보를 업데이트했다면, 그 업데이트된 사실 하나하나가 이벤트다. 백엔드 애플리케이션에서 API 호출을 통해 남기는 로그 하나하나도 이벤트로 볼 수 있다.

### 스트리밍

스트리밍은 이벤트가 발생할 때마다 데이터 소스에서 목적지까지 끊임없이 실시간으로 흘러가는 상태를 뜻한다. 배치와 대비하면 이해하기 쉽다.

| 구분 | 처리 방식 | 예시 |
|------|-----------|------|
| 배치 | 일정 기간 데이터를 모아 한 번에 처리 | 어제 하루 매출 집계 |
| 스트리밍 | 이벤트 발생 즉시 실시간으로 처리 | 사용자 클릭 즉시 추천 반영 |

### 플랫폼

카프카는 데이터 파이프라인의 뼈대가 되는 **중앙 허브**로서의 플랫폼이다. 데이터가 파이프라인처럼 계속 흘러갈 수 있도록 하는 중추적인 역할을 하고, 그 위에서 여러 애플리케이션을 필요에 따라 만들어 나갈 수 있다.

---

## 2. 메시지 큐로 이해하는 카프카

카프카의 세부 구조를 다루기 전, 가장 쉽게 접근하는 방법은 **메시지 큐**로 바라보는 것이다. 메시지 큐는 FIFO로 동작하는 큐 자료구조에 메시지를 쌓아두는 개념이다. 프로듀서가 메시지를 쌓고, 컨슈머가 필요에 따라 그 메시지를 꺼내어 처리한다.

물론 카프카를 단순한 메시지 큐로만 보기엔 부족하다. 다만 처음 접하는 단계에서는 이 정도 이해로도 충분하고, 세부적인 내부 구조는 이후에 별도로 다룰 예정이다.

---

## 3. 카프카는 왜 탄생했는가

카프카를 제대로 이해하려면 **링크드인**이라는 회사가 어떤 상황에 처해 있었는지를 살펴볼 필요가 있다.

### 급격한 성장과 비동기 처리의 필요성

링크드인은 2003년 서비스를 시작한 이후 사용자 수가 폭발적으로 증가했다.

| 연도 | 사용자 수 |
|------|-----------|
| 2004 | 100만 명 |
| 2006 | 500만 명 |
| 2007 | 1,000만 명 |
| 2009 | 5,000만 명 |

이런 성장 속에서 뉴스피드나 광고 시스템처럼 사용자 액션으로부터 **디커플링되어 비동기로 동작**해야 하는 애플리케이션이 늘어났다.

> **디커플링(Decoupling):** 시스템 간의 의존성을 끊어내는 것

사용자가 이직 정보를 업데이트하면 DB 반영은 동기로 처리해야 하지만, 뉴스피드 갱신이나 타겟팅 광고 조정까지 동기로 처리하면 응답 시간이 늘어날 수밖에 없다.

```
사용자 요청
    → DB 저장 (100ms)
    ← 즉시 응답 반환 ✅ "게시 완료!"

        (백그라운드에서 별도로)
        → 뉴스피드 API
        → 광고 API
```

링크드인은 이 문제를 해결하기 위해 ActiveMQ 같은 기존 상용 메시지 시스템을 도입했지만, 관리와 확장이 매우 어렵다는 한계에 부딪혔다.

### 파이프라인의 난립과 신뢰할 수 없는 데이터

사용자 수가 늘어날수록 여러 팀에서 데이터를 필요로 하면서 시스템 간 데이터 파이프라인이 우후죽순으로 생겨났다. 문제는 이 파이프라인들이 그때그때 필요에 맞춰 애드혹하게 구축되었다는 점이다.

| 파이프라인 | 장점 | 단점 |
|------------|------|------|
| 로그 파이프라인 | 확장 용이 | 데이터 유실 발생 |
| Oracle 인스턴스 간 | 빠르고 정확 | 확장 어려움 |
| Hadoop 배치 | 처리량 우수 | 실시간성 부족 |
| 검색 시스템 | 지연 짧음 | DB와 강결합 |

파이프라인이 복잡해질수록 필연적으로 데이터를 신뢰할 수 없게 된다. 실제로 링크드인에서는 비슷한 데이터를 보유한 두 시스템을 점검했더니 서로 다른 값이 나왔고, 확인을 위해 세 번째 시스템을 살펴봤더니 그 값마저 앞의 두 시스템 중 어느 쪽과도 일치하지 않는 일이 벌어졌다.

신뢰할 수 없는 데이터 → 신뢰할 수 없는 보고서 → 신뢰할 수 없는 의사결정. 결국 구성원들이 새로운 걸 만드는 대신 데이터 품질을 바로잡는 데 시간을 쏟아야 하는 상황에 이르렀다.

### 배치의 한계와 실시간 처리의 필요성

링크드인은 Hadoop을 활용해 데이터를 다뤄보려 했지만, 대부분의 애플리케이션이 사용자 액션에 직접 반응해야 하는 실시간 서비스라는 점에서 배치 방식의 한계가 뚜렷했다.

결국 링크드인에는 **비동기적이면서도 실시간으로 빠르게 처리할 수 있는 새로운 무언가**가 필요했다.

### 결단: 처음부터 새로 만들기

이런 문제들이 반복되자 링크드인은 2010년, 기존 상용 시스템을 가져다 쓰는 대신 **데이터 스트림 모델링에 중점을 둔 자체 시스템**을 처음부터 구축하기로 결정한다. 세 명의 엔지니어가 개발을 시작한 것이 바로 아파치 카프카다.

카프카가 자리를 잡은 이후 링크드인의 데이터 흐름은 다음과 같이 바뀌었다.

```
사용자 프로필 수정
    → 이벤트가 카프카에 적재
    → 스트림 프로세싱 레이어가 컨슘 및 표준화
    → 표준화된 데이터를 카프카의 또 다른 파이프라인에 적재
    → 서치 인덱스 / 소셜 그래프 / 추천 시스템 / 데이터 웨어하우스가 각자 컨슘
```

이 모든 과정이 이상적인 상황에서는 **밀리세컨드 단위**로 처리된다.

---

## 4. 카프카의 장점

링크드인이 겪었던 문제들을 되짚어보면 카프카의 장점이 왜 이렇게 설계되었는지 자연스럽게 이해할 수 있다.

| 장점 | 내용 |
|------|------|
| 높은 처리량 | 초당 수백만 개의 메시지 처리, 하루 1조 건 이상의 이벤트 처리 가능 |
| 낮은 지연 | 프로듀싱부터 컨슘까지 수 밀리세컨드 수준 (정상 소비 중 기준) |
| 높은 확장성 | 트래픽이 늘어난 만큼 유연하게 확장 가능 |
| 영속적인 저장 | 일부 노드에 문제가 생겨도 데이터 유실 없이 보관 |
| 고가용성 | 일부 노드 장애가 생겨도 시스템 전체는 정상 동작 |

## 5. 언제 카프카를 쓰면 좋은가

- 사용자 액션으로부터 비동기적으로 동작해야 하는 애플리케이션이 계속 늘어나는 경우. 예약 완료나 구매 완료 후 알림 발송처럼 굳이 동기로 처리하지 않아도 되는 작업이 여기에 해당한다.
- 여러 마이크로서비스와 시스템 사이의 데이터 파이프라인이 계속 늘어나는 경우. 애드혹하게 파이프라인을 뚫기보다 중앙에서 표준화된 형태로 데이터를 주고받을 필요가 생겼을 때다.
- 실시간으로 빠르게 이벤트를 처리하거나 분석해야 하는 경우. 특히 최근 상태값만이 아니라 **상태가 어떻게 변화해왔는지 그 흐름 자체**가 필요한 경우에 유용하다.

## 6. 언제 카프카를 쓰지 말아야 하는가

- **즉시 일관성이 필요한 트랜잭션.** 포인트 차감처럼 정합성이 즉시 보장되어야 하는 작업을 비동기로 처리하면 컨슈머가 DB를 갱신하기 전까지 정합성이 깨질 위험이 있다.
- **복잡한 쿼리나 검색이 필요한 경우.** 카프카는 데이터 파이프라인이지 RDBMS가 아니다. KSQL 같은 방법이 있긴 하지만 일반적인 SQL에 비해 기능이 제한적이다.
- **소규모 트래픽의 스타트업 초기 단계.** 카프카 도입에는 운영 관리 오버헤드, 추가 인프라 비용, 디버깅 난이도가 따른다. 트래픽이 실제로 늘어난 시점에 도입을 고려하는 편이 합리적이다.

---

## 7. 실무에서 함께 따라오는 개념들

분산 환경에서 여러 마이크로서비스가 동일한 이벤트를 활용해야 하는데 트랜잭션으로 묶기 어려운 경우, **사가 패턴(Saga Pattern)** 이나 **아웃박스 패턴(Outbox Pattern)** 같은 해결책이 존재한다. 아웃박스 패턴을 구현하는 여러 방식 중 하나로 카프카를 활용하는 방법도 있다.

**이벤트 소싱(Event Sourcing)** 이라는 용어도 함께 등장하는데, 이벤트를 계속 적재하고 전달한다는 정도로 우선 이해하면 충분하다.

---

## 마무리

카프카는 오픈소스, 분산, 이벤트, 스트리밍, 플랫폼이라는 다섯 가지 키워드로 요약된다. 이 모든 특징은 링크드인이 겪었던 실제 문제, 즉 비동기 처리의 필요성, 파이프라인 난립으로 인한 데이터 신뢰성 붕괴, 배치의 한계에서 비롯된 실시간 처리 요구에서 비롯되었다.

기술의 탄생 배경을 이해하고 나면 카프카를 언제 도입하고 언제 신중해야 하는지에 대한 감각이 훨씬 명확해진다. 무조건 카프카를 쓰는 것이 아니라, 카프카가 풀도록 설계된 문제와 지금 마주한 문제가 실제로 닮아 있는지를 먼저 따져보는 태도가 필요하다.

> 다음 글에서는 카프카의 핵심 내부 구조인 브로커, 토픽, 파티션, 오프셋을 다룬다.
